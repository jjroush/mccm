What this builds

Three LEDs on a breadboard mirror the aggregate state of all your Claude Code sessions. mccm's hooks already write every session's status to ~/.claude/mccm/state.json; a new mccm led daemon watches that file and streams a one-byte status over USB serial to an ESP32-C6, whose firmware switches the LEDs.

Serial protocol

One ASCII byte plus newline per update. The host resends the current state every 2 seconds as a heartbeat, so the board converges after a replug with no handshake. The firmware ignores unknown bytes (including the \n), keeping the protocol forward-compatible.

ByteSession stateLEDs
Rat least one session needs helpred on
Yat least one session inactiveyellow on
Gall sessions actively workinggreen on
Nno live sessionsall off
Design note The colors match the macOS menu bar icon exactly: green / yellow / red, and the same priority order — needs-help beats inactive beats active. The firmware also accepts B as a legacy alias for Y (the yellow LED started life as a blue one), so either side of the rename still interoperates.

Board discovery — MuseLab nanoESP32-C6

The board (designed by wuxx/MuseLab, ESP32-C6-WROOM-1 module) has two USB-C ports, and the choice between them was the first real design decision:

Port (silkscreen)What it isUSB VID:PIDVerdict
ESP32C6 — next to RST button The chip's built-in USB-Serial-JTAG peripheral (GPIO12/13 internally) 303a:1001 (Espressif) ✅ Used. One cable flashes the firmware and carries status bytes.
CH343 — next to BOOT button WCH CH343P USB-UART bridge → UART0 (GPIO16/17) 1a86:55d3 (WCH) Fallback only. Its auto-reset circuit can reboot the chip when the port opens.

Other facts that shaped the design:

  • Onboard WS2812 RGB LED on GPIO8 — could show all three colors with zero wiring, but needs a timing-critical one-wire protocol driver (RMT peripheral) instead of three set_high() calls, and GPIO8 is a strapping pin. Discrete LEDs are the better first project; the WS2812 is a nice follow-up.
  • BOOT button = GPIO9, RST = chip enable. If a bad flash ever kills the USB port: hold BOOT, tap RST → ROM bootloader → recovery flash.
  • Both ports power the board; GPIO logic level is 3.3 V.

Pin choice — why GPIO18 / 19 / 20

The embedded nuance: on a microcontroller, almost every pin has a side job. "Plain GPIO with no reset-time meaning" is the thing you're shopping for. The elimination on the ESP32-C6:

PinsSide jobRisk if used
GPIO4, 5, 8, 9, 15Strapping pins, sampled at reset (8/9 select boot mode)An LED's pull could change the boot mode — board won't start until rewired
GPIO12, 13Native USB D− / D+Reusing them kills the serial link this whole project runs on
GPIO16, 17UART0 ↔ CH343 bridgeLoses the backup serial/flash path
GPIO24–30SPI flash inside the moduleNot even broken out
GPIO18, 19, 20None— and they're physically adjacent, two pins from a GND

Schematic

Each GPIO sources current through a resistor and LED to ground (active-high). Long LED leg (anode) toward the resistor; short leg / flat side (cathode) toward GND.

Breadboard steps

  1. Seat the board across the breadboard's center channel (or beside it with jumpers, if headers aren't soldered yet).
  2. Jumper the board's GND (bottom row, 2nd pin from the USB end) to the breadboard's blue (−) rail.
  3. For each color: GPIO jumper → 330 Ω resistor → LED long leg; LED short leg → (−) rail.
  4. Plug USB-C into the ESP32C6 port. On boot the firmware flashes all three LEDs together three times (2 s per flash) as a wiring self-test.

Electrical reasoning

  • GPIOs swing 0 → 3.3 V. Red and yellow LEDs drop ~2.0–2.1 V, so 330 Ω passes (3.3 − 2.1) / 330 ≈ 3.6 mA — nicely visible; kit greens are usually the same low-voltage chemistry. Blue and white LEDs drop ~3.0 V+, leaving ~1 mA on 330 Ω — nearly invisible, which is why this project's blue became yellow.
  • The resistor is not optional. An LED is a diode, not a resistor — wired bare it draws whatever the pin can deliver and cooks the LED, the pin, or both. Safe budget: ~10 mA per pin (the C6 can push more; no reason to).
  • Resistor on either leg works — it's a series loop.

Firmware decisions (esp32-firmware/ · esp-hal 1.1.1, no_std)

  • RISC-V perk: the C6 builds on plain stable Rust (riscv32imac-unknown-none-elf) — no espup/Xtensa toolchain. The rust-toolchain.toml auto-installs the target on first build.
  • Never writes to USB. A blocking TX write on USB-Serial-JTAG stalls forever if no host program is reading the port — and mccm led only sends. RX is safe: read_byte() is non-blocking and just returns WouldBlock.
  • All three LEDs on = "no host daemon." That's the state after boot and where the board falls back if the 2 s heartbeat goes silent for 10 s — a dead daemon reads as obviously-disconnected, never as a stale "all good".
  • Boot self-test flashes all three LEDs together three times (2 s per flash) so wiring is verified before any software enters the picture.
  • esp_app_desc!() embeds the metadata block the ESP-IDF second-stage bootloader expects; newer espflash refuses images without it.
  • The usb_serial_jtag driver sits behind esp-hal's unstable cargo feature — required, documented in Cargo.toml.
  • Watchdogs: nothing to feed — esp_hal::init() disables them by default.

Host daemon decisions (src/led.rs · mccm led)

  • Auto-detection by USB vendor ID, in preference order: 303a Espressif native USB → 1a86 WCH CH343 → 10c4 Silicon Labs CP210x → any USB serial device. Override with mccm led --port /dev/cu.usbmodem….
  • macOS nuance: prefers the callout device (cu.*) over dial-in (tty.*) — cu.* opens without waiting for carrier-detect.
  • Send-on-change + 2 s heartbeat, polling state.json every 500 ms. ~1 byte/sec of traffic, and the board self-corrects after any replug.
  • Reconnect loop: unplugging the board drops the daemon back to scanning every 2 s until it reappears — leave it running unattended.
  • The aggregate logic (red beats yellow beats green) moved from menubar.rs into state.rs, so the menu bar icon and the LEDs can never disagree.

Run it

# 1. Wire the breadboard (schematic above), plug into the ESP32C6 port
# 2. Flash:
cd ~/Developer/cc-middle-manager/esp32-firmware
cargo run --release          # espflash flash --monitor; Ctrl-C exits monitor

# 3. Install + run the daemon:
cd ~/Developer/cc-middle-manager
cargo install --path .
mccm led                     # auto-detects the board, reconnects on replug
